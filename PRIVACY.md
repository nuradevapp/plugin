# Privacy

This document describes what data the **Nura Dev Claude Code plugin** sends out of your machine and what it does **not** send. It covers the plugin code in this repository — relay-side processing (voice transcription, push delivery) is operated by Nura Dev as a managed service and is documented at <https://nuradev.app/privacy>.

## Connection

The plugin opens a single TLS WebSocket to `wss://relay.nuradev.app` for the lifetime of your Claude Code session. There is no other outbound connection from the plugin. Self-hosted relays are not supported.

## What the plugin sends out

### Identifiers

| Sent | Source | Purpose |
|---|---|---|
| `pluginToken` | Random opaque string generated by the relay during pairing, stored locally at `~/.nuradev/plugin-token.json` (mode `0600`) | Authenticates this directory to the relay on reconnect, so you don't re-pair every session |
| `sessionId` | Assigned by the relay per Claude Code session | Routes voice messages to the right Claude Code instance |
| `deviceName` | Your machine's hostname, sent only on the first pairing request | Helps you identify the right machine in the phone app's session list |

### Tool activity (`activity_event`, sent at every tool start and end)

For every tool Claude Code runs, the plugin sends:

- The tool name (`Read`, `Edit`, `Write`, `Bash`, `Grep`, etc.)
- A short human-readable summary, e.g. `Read auth.ts` or `Bash: npm install` (Bash commands are truncated to the first 40 characters)
- A monotonically-increasing ID and a start/end phase
- A timestamp

The summary uses **basenames only** for file paths, never absolute paths or directory listings.

### Hook status pings (`status`)

After every tool call, a one-line status string ("Edit auth.ts", "Done.", etc.) is forwarded so your phone shows current activity. The string is identical to what the activity feed already contains — no additional information.

### Replies from Claude (`reply`, `reply_with_detail`)

When Claude calls the plugin's `reply` tool to respond to a voice message, the plugin forwards Claude's text reply to the relay so it can be read aloud on your phone. `reply_with_detail` includes a longer-form version shown when you tap the message card.

The plugin only sends what Claude explicitly passes to the `reply` tool. It does not scrape Claude Code's transcript or include surrounding context.

### Permission prompts (`permission_request`)

When Claude wants to run a tool that needs approval, the plugin forwards:

- The tool name (e.g. `Bash`, `Write`, `Edit`)
- Claude Code's human-readable description of the call (the same text shown in the local terminal dialog)
- A 200-character preview of the tool's arguments

The arguments preview is sent verbatim. For Bash this is the command string; for Write/Edit it's the file path and the first ~200 chars of the contents Claude would write. Sensitive content can appear here if Claude is editing a file that contains it. Treat the relay channel and your paired phones with the same trust you give the local terminal dialog.

### Task updates (`task_update`)

When Claude uses `TaskCreate` or `TaskUpdate`, the plugin sends the task ID, title, description, and status so the phone app can render a persistent task board.

### Inline images (in replies only)

When Claude includes an image in a reply (e.g. a screenshot it generated), the image is base64-encoded and included in the reply payload. The plugin never reads images from disk on its own.

### Lifecycle and meta

- `register_plugin` — sent once on connect; carries the `sessionId` for reconnect resumption.
- `request_pairing_code` — sent when no token exists for the current working directory.
- `thinking` — sent when a voice message arrives so the phone can show an indicator.
- `activity_clear` — sent on plugin shutdown to clear the phone's activity board.

## What the plugin does NOT send

The plugin **never** sends:

- File contents (only basenames inside summaries; never the bytes of a read or written file beyond the 200-char `input_preview` for permission prompts described above)
- Claude Code's full conversation transcript
- The diffs Claude generates beyond the high-level reply text
- Environment variables, API keys, or shell history
- Git history, working directory listings, or repository metadata
- Anything from any directory other than the one the plugin is paired against

## What the relay sends back to the plugin

| Inbound | Purpose |
|---|---|
| `pairing_code` | Short code displayed in your terminal during initial pair |
| `paired` | Confirms pairing; carries the `pluginToken` to be persisted |
| `registered` | Confirms reconnect; carries the assigned `sessionId` |
| `message` | A voice message from a paired phone (transcribed text, plus optional image or file attachment) |
| `permission_verdict` | An `allow`/`deny` answer to a permission request the plugin previously sent |
| `app_disconnected` / `app_reconnected` | Phone app connection state changes |
| `command` | A slash-command string entered on the phone (e.g. `/clear`) |

Inbound messages are gated by the relay's pairing system: only phones the user has explicitly paired through the pairing-code flow can push messages or permission verdicts into the session. The pairing code expires after 5 minutes.

## Local storage

| Path | Mode | Contents |
|---|---|---|
| `~/.nuradev/plugin-token.json` | `0600` | An array of `{ directory, pluginToken, pluginTokenId, sessionId }` records, one per working directory the user has paired |
| `/tmp/nuradev-session.<ppid>` | default | The current session ID; read by the hook CLI to attach activity events to the right session |

Removing `~/.nuradev/plugin-token.json` un-pairs the machine; the next launch will request a fresh pairing code. Tokens revoked on the relay side cause the WebSocket to close with code `4401`, which prompts the plugin to delete the file automatically and re-pair.
